## 0. Authority, Objective and Agent Instructions

This document replaces earlier high-level ideas, technology comparisons, and incomplete feature lists. It is the single source of truth for designing, building, testing, deploying, documenting, and handing over the School Management System.

Build a modern, secure, highly interactive, school-first web application that replaces scattered paper registers, manual fee books, attendance sheets, admission files, spreadsheets, class diaries, certificates, basic accounts registers, and follow-up notebooks with one integrated system.

The system must make it realistic for **one authorised operator** to manage routine school operations efficiently, while still supporting teachers, accountants, librarians, parents, students, and other staff through controlled access.

### Non-negotiable agent rules

1. Do not build a static mock-up, a demo-only dashboard, or disconnected forms.
2. Build working frontend, backend, database migrations, validation, permissions, audit logs, reports, imports, printable documents, automated tests, and deployment configuration.
3. Do not ask the operator to enter the same information twice in different modules. One canonical data record must feed every related register and report.
4. Never hard-delete official records such as admissions, invoices, payments, attendance, marks, payroll, certificates, accounting entries, stock movements, or audit records.
5. Every high-impact change must be traceable: who changed it, when, why, what the previous value was, and what the new value is.
6. The platform must work smoothly on desktop first, then tablet and mobile. School office work is primarily desktop-oriented.
7. Use clean, maintainable production code. No placeholder data, fake API calls, mock payment success, or unconnected UI in the final build.
8. Use the selected architecture below. Do not introduce microservices, Kubernetes, or additional frameworks unless they solve a documented requirement.
9. Build feature flags for optional or later modules. The initial system must remain understandable and fast for a small school.
10. The system must be branded throughout as stated in Section 2.

## 1. Product Vision and Measurable Success

### 1.1 Product vision

Create a “single operating desk” for a school. From one dashboard, an authorised operator should be able to:

- admit and enrol students;
- maintain student, guardian, staff, class, subject, and academic-year records;
- mark and correct attendance;
- generate fees, collect payments, issue receipts, track dues, and record concessions;
- maintain cash, bank, expense, income, payroll, and accounting records;
- produce report cards, certificates, ID cards, transfer certificates, notices, letters, and registers;
- manage library records, visitor logs, inward/outward dispatch, inventory, and school assets;
- communicate with parents and staff;
- search and retrieve a student, receipt, register entry, document, certificate, or transaction in seconds;
- see what requires action today without checking many physical files.

### 1.2 Success criteria

The first production release is successful only when all of the following are true:

- A student can be admitted, assigned to a class and section, charged the correct fees, issued a receipt, marked present/absent, entered in examinations, promoted, transferred, or archived without data duplication.
- A full day’s attendance for a class can be recorded in less than five minutes on a desktop.
- A standard fee payment can be collected and a verified receipt can be printed or sent in less than two minutes.
- The system can generate printable A4 register formats for attendance, admissions, fee collection, library, visitors, inward/outward dispatch, and stock.
- Every payment, refund, fee adjustment, marks change, attendance correction, certificate, and accounting voucher has an immutable audit trail.
- Daily backup health is visible to the owner/operator.
- A new operator can use the system after guided onboarding without technical training.
- No critical school record depends on a spreadsheet, local browser storage, or an individual employee’s personal device.

## 4. Non-Negotiable Technology Decisions

### 4.1 Architecture

Build a **modular monolith**. It is simpler, cheaper, safer, and easier for one school to maintain than a distributed architecture.

- **Frontend:** Next.js with TypeScript, component-driven UI, server-safe rendering, accessible forms, and a responsive application shell.
- **Backend:** Django with Django REST Framework.
- **Database:** PostgreSQL.
- **Back-office administration:** Django Admin, enhanced only where it adds real operational value.
- **Background jobs:** Celery with Redis.
- **File storage:** S3-compatible object storage in production; local development storage only for local development.
- **Authentication:** secure HttpOnly same-site session cookies with CSRF protection. Do not expose long-lived tokens to browser JavaScript.
- **API documentation:** OpenAPI generated from the API and maintained as part of CI.
- **PDF generation:** HTML print templates rendered by a controlled server-side PDF engine. Validate all official templates with real A4 print tests.
- **Deployment:** Docker Compose for local development and initial production deployment; separate staging and production environments.
- **CI/CD:** GitHub Actions or equivalent pipeline for linting, testing, migration checks, build, and deployment.
- **Error tracking:** Sentry or equivalent.
- **Email/SMS/WhatsApp:** provider adapters behind a single notification interface. Never spread provider-specific code through business modules.

### 4.2 Why PostgreSQL is mandatory

School operations are highly relational and audit-sensitive: students have many guardians, years of enrolments, attendance records, invoices, receipts, subjects, results, documents, and certificates. Financial records must reconcile. PostgreSQL provides the transactions, constraints, joins, locking, and reporting reliability required.

Do not use MongoDB as the primary system of record.

### 4.3 No hard-delete policy

Records use one of the following controlled states instead:

- draft;
- active;
- submitted;
- approved;
- finalised;
- cancelled;
- reversed;
- archived;
- superseded;
- inactive.

Deletion may be allowed only for accidental, unreferenced drafts by a privileged role. The deletion event itself must be logged.

## 5. Tenant and School Model

Build the database as **single-school-first but multi-school-ready**.

- Every operational table must include `school_id`.
- Add `campus_id` where a record may belong to a campus.
- Enforce row-level school filtering in backend querysets and permission checks.
- Never trust a school identifier submitted by the browser without validating that the authenticated user belongs to that school.
- Version 1 user interface may expose one active school only, but the schema must prevent future cross-school leakage.

## 6. User Roles and Permission Model

### 6.1 Roles

1. **Owner / Super Administrator**  
   Full school control, settings, backup visibility, user administration, financial closure, audit access, and emergency controls.

2. **School Operator**  
   The one-person operations role. Can handle admissions, attendance, fees, registers, certificates, notices, and routine reports. Cannot silently erase financial or audit history.

3. **Principal / Academic Administrator**  
   Academic configuration, result approval, timetable approval, student promotion, certificates, discipline, and reports.

4. **Accountant**  
   Fee plans, receipts, payment reconciliation, cash/bank, expenses, vouchers, payroll preparation, and finance reports.

5. **Teacher**  
   Assigned class/subject attendance, marks entry, class diary, homework or notices if enabled. Cannot view unrelated students’ confidential records.

6. **Librarian / Storekeeper**  
   Library and/or stock operations only.

7. **Gate / Reception Staff**  
   Visitor register, inward/outward register, enquiries, basic admission lead capture, and controlled student lookup.

8. **Parent / Guardian**  
   View only their linked student(s): attendance, fees, receipts, notices, timetable, results, documents explicitly released by the school.

9. **Student**  
   Optional and limited: timetable, notices, released results, library status, and learning links.

### 6.2 Permission implementation

Do not use role names alone as authorisation. Implement granular permissions such as:

- `student.view`, `student.create`, `student.edit`, `student.archive`;
- `fee.plan.manage`, `fee.collect`, `fee.refund.request`, `fee.refund.approve`;
- `attendance.mark`, `attendance.correct`, `attendance.finalise`;
- `exam.marks.enter`, `exam.marks.approve`, `exam.results.publish`;
- `ledger.view`, `voucher.create`, `voucher.approve`, `period.close`;
- `register.print`, `register.export`;
- `audit.view`, `user.manage`, `settings.manage`.

Sensitive actions require re-authentication or an elevated confirmation:

- refund;
- receipt cancellation;
- marks amendment after publication;
- salary release;
- financial period close;
- user privilege escalation;
- data export containing confidential fields.

## 7. Core Data Principles

### 7.1 Identifiers

Use internal UUID primary keys. Use separate human-readable identifiers for official use:

- admission number;
- current academic-year roll number;
- employee number;
- invoice number;
- receipt number;
- voucher number;
- certificate number;
- library accession number;
- stock issue number;
- visitor pass number.

All human-readable sequences must be configurable, unique within the correct scope, and never reused.

### 7.2 Historical records

Never overwrite academic history. A student’s current class is a convenience field only. The authoritative history is maintained through enrolment records.

### 7.3 Canonical source data

A register is normally a **view or printout of canonical data**, not a separate duplicate record. For example:

- attendance register is generated from attendance sessions and entries;
- fee collection register is generated from receipts and payment allocations;
- admission register is generated from admissions and enrolments;
- library accession register is generated from book-copy records;
- payroll register is generated from payroll runs and payslips.

Only standalone operational logs use their own entries, such as visitor, inward/outward dispatch, incident, class diary, and complaint registers.

### 7.4 Audit event standard

Every audit event includes:

- event ID;
- school and campus;
- actor user;
- role at the time of action;
- action type;
- entity type and ID;
- previous data snapshot or field-level changes;
- new data snapshot or field-level changes;
- reason / correction note where required;
- timestamp;
- IP/device context where practical;
- request/correlation ID.

Audit records are append-only and not editable from the standard user interface.

## 9. End-to-End Workflows

### 9.1 Initial school setup wizard

The first-run wizard must guide an owner through:

1. school name, logo, address, contact, letterhead, and authorised signatory;
2. academic year, terms, holidays, working days, and attendance rules;
3. classes, sections, capacities, subjects, periods, and rooms;
4. fee heads, fee plans, payment methods, late fee rules, receipt formats, and initial balances;
5. staff roles, user accounts, payroll defaults, leave types;
6. grading scales, exam terms, and report-card format;
7. document types, certificate templates, numbering sequences, and communication settings;
8. backup configuration and administrator recovery details.

No module should expose blank, confusing screens without onboarding guidance.

### 9.2 Admission workflow

`Enquiry → Application → Document upload → Verification → Approval / Rejection / Waitlist → Student record → Guardian linking → Academic enrolment → Class/section allocation → Fee assignment → Initial payment → Receipt → Student ID / admission register entry`

Requirements:

- support walk-in and online applications;
- application number distinct from admission number;
- configurable mandatory documents;
- verify documents with status, verifier, and remark;
- prevent accidental duplicate students using name, date of birth, guardian phone, and prior admission number checks;
- create admission register entry automatically after approval;
- issue a welcome notice, fee receipt, and optional ID card;
- allow drafts and incomplete applications without treating them as admitted.

### 9.3 Academic-year rollover and promotion workflow

`Prepare next year → configure new classes/fees → review results/dues → bulk promotion / retention / transfer / graduate → create new enrolments → assign class/section/roll number → carry permitted balances → archive old-year outputs`

Requirements:

- never mutate the old student-enrolment record;
- generate promotion report before finalisation;
- flag students with unpaid balances, incomplete documents, or pending decisions;
- require review before batch promotion;
- support repeat, transfer, withdraw, graduate, and discontinued outcomes;
- freeze old-year marks, reports, and registers once the closure is approved.

### 9.4 Attendance workflow

`Create session → mark attendance → save draft → submit/finalise → correction request if needed → register/report generation → parent notification rules`

Requirements:

- class-wise grid with keyboard navigation;
- default present with exceptions, and option to default absent with exceptions;
- bulk mark, bulk leave, search, and filter;
- late/half-day/leave reasons;
- one-click print attendance register;
- daily absence alerts using configured rules;
- correction requires reason and, after finalisation, approval if policy demands it.

### 9.5 Fee workflow

`Configure fee plan → assign fee plan to enrolment → generate due items/invoices → collect payment → allocate payment → issue receipt → post accounting → notify guardian → monitor defaulters → adjust/refund/reverse if authorised`

Requirements:

- cash, bank transfer, UPI, card, cheque, and online gateway methods;
- partial payments, advance payments, instalments, concessions, scholarships, late fees, waivers, adjustments, and refunds;
- cash drawer closing and daily collection summary;
- printable receipt with QR verification;
- no duplicate receipts and no payment posting without a unique reference;
- automated reminders only after approved communication configuration;
- defaulter reports by class, due date, fee head, and guardian contact.

### 9.6 Expense and accounting workflow

`Create voucher → attach bill/document → review → approve/post → payment → journal posting → cash/bank update → report`

Requirements:

- expense, income, journal, contra, and payment voucher types;
- configurable approval threshold;
- receipt/invoice attachment;
- immutable posted vouchers;
- daily cash closing;
- bank reconciliation support;
- financial reports filtered by date range, campus, and account head.

### 9.7 Examination workflow

`Configure term/exam → assign classes/subjects → schedule → enter marks → validate → moderation → approve → publish → issue report cards → correction/re-evaluation through controlled process`

Requirements:

- marks entry grid with autosave;
- export/import marks sheets with validation;
- validation of maximum marks, absent/exempt states, missing entries, and grading rules;
- draft results not visible to parents/students;
- finalised results require principal/authorised approval;
- report cards must show issue date, document number, QR verification, and signatory fields.

### 9.8 Staff and payroll workflow

`Create staff record → assign role/user account → configure salary → attendance/leave → draft payroll → review → approve → pay → issue payslip → ledger posting`

Requirements:

- separation between staff profile and user login;
- protected bank and salary information;
- payroll cut-off and locking;
- payslip PDF;
- leave balances and attendance-adjustment rules;
- no direct editing of paid payroll.

### 9.9 Library workflow

`Add title → add physical copies/accession numbers → issue → return/renew → fine/lost/damaged action → reports and accession register`

### 9.10 Inventory workflow

`Create item → receive goods → movement record → issue to staff/class → stock count → adjustment with reason → asset tracking / disposal`

### 9.11 Certificates and official documents workflow

`Select student/staff → choose template → prefill verified data → review → issue → number and QR verification → archive issuance record`

Initial templates:

- student ID card;
- bonafide certificate;
- transfer certificate;
- school leaving certificate;
- character certificate;

- attendance certificate;
- report card;
- library clearance;

## 12. API and Backend Standards

### 12.1 API structure

Use versioned REST endpoints:

- `/api/v1/auth/...`
- `/api/v1/students/...`
- `/api/v1/admissions/...`
- `/api/v1/attendance/...`
- `/api/v1/fees/...`
- `/api/v1/accounting/...`
- `/api/v1/exams/...`
- `/api/v1/staff/...`
- `/api/v1/library/...`
- `/api/v1/inventory/...`
- `/api/v1/registers/...`
- `/api/v1/reports/...`
- `/api/v1/settings/...`

### 12.2 Endpoint requirements

Every collection endpoint must support, where relevant:

- pagination;
- sort;
- validated filters;
- search;
- date range;
- export request;
- permission enforcement;
- school/campus scope enforcement.

Every mutation endpoint must:

- validate payload server-side;
- return clear field-level errors;
- create audit event;
- enforce state transition rules;
- be idempotent where payment/import/webhook retries are possible;
- use database transactions for multi-record operations;
- never rely on client-side validation alone.

### 12.3 API error standard

Return predictable JSON:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Please correct the highlighted fields.",
    "fields": {
      "admission_number": ["This admission number already exists."]
    },
    "request_id": "..."
  }
}
```

Do not return raw stack traces or database errors to users.

### 12.4 Import APIs

Use controlled imports for:

- students;
- guardians;
- staff;
- attendance;
- fee opening balances;
- books;
- inventory;
- marks.

Every import needs:

1. downloadable template;
2. preview and validation report;
3. duplicate detection;
4. row-level errors;
5. dry run;
6. explicit confirmation;
7. audit record;
8. reversible import strategy where feasible.

## 13. Security, Privacy and Data Governance

### 13.1 Security baseline

Implement:

- HTTPS only in production;
- secure, HttpOnly, same-site cookies;
- CSRF protection;
- password hashing using framework-recommended secure algorithms;
- login throttling and lockout controls;
- password reset with short-lived signed tokens;
- optional MFA for owner and finance roles;
- RBAC with server-side enforcement;
- content security policy;
- secure file upload validation and malware scanning integration point;
- dependency scanning;
- secret storage only through environment/secret manager;
- rate limits for login, password reset, public verification, and payment callbacks;
- separate staging and production credentials;
- regular database backups and restore tests.

### 13.2 Child and family data protection

Treat personal data as confidential:

- minimise collection;
- collect and store consent records;
- restrict health, identity, bank, salary, and disciplinary data;
- log exports and sensitive document downloads;
- never place confidential student data in public URLs;
- hide data from roles that do not need it;
- keep parent portal limited to linked students;
- provide retention/archival configuration;
- prevent AI or third-party messaging providers from receiving more data than necessary.

### 13.3 File management

- store files outside the application container;
- use opaque object keys, not student names in public paths;
- sign download URLs with expiry;
- record document type, owner entity, uploader, timestamp, verification status, and access restrictions;
- generate thumbnail/preview safely where enabled;
- reject executable files and unsafe file types.

### 13.4 Backups and recovery

Implement:

- automatic daily PostgreSQL backup;
- file-storage backup policy;
- backup status dashboard;
- encrypted backup storage;
- documented restore runbook;
- restore-test schedule;
- recovery objective and recovery point targets configured by deployment environment;
- export of critical annual records before academic-year closure.

## 14. Notifications, Communication and Smart Assistance

### 14.1 Notification subsystem

Create provider-neutral channels:

- in-app;
- email;
- SMS;
- WhatsApp, where approved/configured;
- print queue / physical notice.

Notification events include:

- admission confirmation;
- fee receipt;
- due-date reminder;
- absent student;
- leave decision;
- examination result published;
- notice/circular;
- library overdue;
- certificate issued;
- payment/refund status;
- staff payroll availability.

Every delivery must log:

- channel;
- recipient;
- template;
- payload reference;
- provider message ID;
- delivery status;
- retry count;
- failure reason.

### 14.2 Smart but safe operator assistance

Add an optional internal “Office Assistant” after the core workflows are stable.

It may:

- explain what requires action today;
- suggest pending records;
- prepare a draft notice;
- help locate a record;
- generate a filtered report request;
- explain a workflow step;
- flag possible duplicate student entries;
- identify unbalanced accounting draft entries;
- provide read-only summaries.

It must never independently:

- alter attendance;
- change marks;
- collect or refund money;
- issue certificates;
- post accounting entries;
- approve payroll;
- delete data;
- communicate externally without explicit user review and confirmation.

All AI-generated outputs must be visibly labelled as drafts. Do not send sensitive student data to external AI services without a documented privacy review and explicit administrator consent.

## 15. Reporting, Exports and Verification

### 15.1 Core reports

Build first:

- admissions by period/class/status;
- student roll list;
- student contact directory with restricted export;
- attendance daily/monthly/class/individual;
- absentee and leave summaries;
- fee collection daily/monthly/fee-head/class;
- outstanding dues/defaulters;
- concessions, adjustments, refunds, and cancelled receipt report;
- cash book, bank book, day book, ledger, trial balance, income/expenditure;
- staff directory, attendance, leave, payroll, and salary reports;
- examination marks analysis, grade distribution, subject performance, report-card issue list;
- library stock, issued, overdue, fine, lost/damaged;
- inventory stock, issue, low-stock, asset register;
- register exports;
- audit activity report;
- backup and notification delivery health report.

### 15.2 Export rules

- PDF, CSV, and XLSX where appropriate;
- all exports must include report name, filter scope, generated timestamp, generated-by, and school name;
- sensitive exports require permission and audit log;
- respect user-visible column permissions;
- do not expose health/bank/discipline data in general exports.

### 15.3 Public document verification

For receipts, certificates, report cards, and ID cards, generate a QR code that opens a limited public verification page.

The public page must show only:

- document type;
- document number;
- school;
- issue date;
- validity status;
- limited recipient name masked where policy requires.

Never expose full student profiles, marks, guardian contacts, or internal notes publicly.

## 16. Testing and Quality Gates

### 16.1 Automated testing

Implement:

- unit tests for all core business rules;
- API tests for permissions, validation, state transitions, and error formats;
- database constraint tests;
- integration tests for fee receipt to ledger posting;
- integration tests for promotion and enrolment history;
- integration tests for attendance correction;
- end-to-end browser tests for admission, fee collection, attendance, marks publication, certificate issuance, and register print flows;
- regression tests for receipt numbering and audit immutability;
- security tests for cross-school access denial and role access denial;
- import validation tests;
- backup/restore smoke test in staging.

### 16.2 Mandatory test scenarios

At minimum, prove:

1. One guardian linked to two students.  
2. A student moves to a new academic year without losing old roll number, class, fees, attendance, or marks.  
3. A partial payment pays selected dues and leaves correct outstanding balance.  
4. One receipt is cancelled through reversal without reuse of receipt number.  
5. A refund produces correct accounting reversal.  
6. A posted journal entry cannot be edited directly.  
7. A marks correction after publication requires an audit reason.  
8. A teacher cannot access unrelated student records.  
9. A parent cannot access another family’s records.  
10. A user from another school cannot access any data outside their school.  
11. Attendance cannot be recorded twice for the same student/session.  
12. Register prints reflect canonical data and include correct filters.  
13. A failed notification delivery is visible and retryable.  
14. Backup health failure appears on the owner dashboard.  
15. A file upload rejects unsafe types.  

### 16.3 Code quality standards

Require:

- typed frontend code;
- backend linting and formatting;
- migration review;
- no N+1 query patterns in large lists;
- server-side pagination;
- structured logging;
- meaningful commit messages;
- dependency lockfiles;
- environment example files without secrets;
- code comments only where logic is non-obvious;
- no dead code or unused feature flags.

## 18. Data Migration and Go-Live Procedure

### 18.1 Before migration

- collect existing paper registers, spreadsheets, fee books, student files, staff records, library accession lists, and stock lists;
- define clean import templates;
- map old fields to new fields;
- identify missing dates, duplicates, inconsistent spellings, and incomplete contact records;
- establish a unique admission number strategy before import;
- decide which legacy records are entered as historical summaries and which require detailed row-level data.

### 18.2 Migration sequence

1. school settings and academic years;  
2. classes/sections/subjects/staff;  
3. students and guardians;  
4. active enrolments and roll numbers;  
5. opening fee balances;  
6. opening cash/bank/ledger balances;  
7. staff salary structures;  
8. library titles/copies;  
9. inventory/assets;  
10. selected historical documents and reports.  

### 18.3 Cutover checklist

- freeze old manual entries at agreed cutover date;
- reconcile fee total and cash/bank balance;
- verify student count by class;
- verify staff count;
- verify current due totals;
- generate trial balance;
- print/register sample verification;
- create initial backup;
- train operator;
- activate help and incident process;
- maintain read-only access to legacy files for a defined period.

## 23. Agent Execution Protocol

1. Read this document fully.  
2. Create a detailed implementation checklist mapped to each phase.  
3. Set up the repositories, containers, CI, database, authentication, roles, audit, and design system.  
4. Implement one phase at a time.

8. At the end of each phase, produce:
   - completed features;
   - database changes;
   - API changes;
   - test results;
   - known limitations;
   - next-phase dependencies.
9. Never replace essential implementation with placeholders merely to make the interface look complete.  
10. Preserve the product goal: one reliable, secure, intelligent school operating system that reduces manual work instead of transferring manual work into many browser forms.
